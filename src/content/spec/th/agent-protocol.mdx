# โปรโตคอลของเอเจนต์

**เวอร์ชัน 1.2** โปรโตคอลนี้นิยามว่าเอเจนต์เขียนโค้ด AI MUST ประพฤติอย่างไรเมื่อทำงานกับ Deep Work Plan คำสำคัญ MUST, SHOULD และ MAY เป็นไปตาม RFC 2119

> **เพิ่มเติมใน v1.2** การเพิ่มเติมสองอย่าง ไม่มีการเปลี่ยนแปลงที่ทำลายความเข้ากันได้: (1) แพลตฟอร์มเอเจนต์อัตโนมัติ (OpenClaw, Hermes) เข้าร่วมในตารางเอเจนต์ที่รองรับ; (2) ส่วน**โปรไฟล์การดำเนินงาน**นิยามการดำเนินงานแบบไม่มีผู้ดูแล — อำนาจจำกัด ชั้นสถานะบังคับ เงื่อนไขหยุด และ scheduled continuation

## เอเจนต์ที่รองรับ

ระเบียบวิธีนี้ MUST รองรับเอเจนต์เขียนโค้ด AI ต่อไปนี้ เอเจนต์ในอนาคตที่อ่าน markdown และสามารถดำเนินการ tool call MAY ถูกเพิ่มโดยไม่มีการเปลี่ยนแปลงที่ทำลายความเข้ากันได้

| เอเจนต์ | ข้อตกลง config แบบ native | Command prefix |
|-------|--------------------------|----------------|
| **Claude Code** | `.claude/` (symlink ไปยัง `.agents/`) | `/` (native slash commands) |
| **Cursor** | `.cursor/rules/*.mdc` อ้างอิง `AGENTS.md` | `#` หรือข้อความธรรมดา |
| **OpenAI Codex** | `.codex/` อ้างอิง `AGENTS.md` | `#` หรือข้อความธรรมดา |
| **Google Gemini** | `.gemini/` อ้างอิง `AGENTS.md` | `#` หรือข้อความธรรมดา |
| **GitHub Copilot** | `.github/copilot-instructions.md` อ้างอิง `AGENTS.md` | `#` หรือข้อความธรรมดา |
| **Antigravity** | `.antigravity/` อ้างอิง `AGENTS.md` | `#` หรือข้อความธรรมดา |
| **OpenClaw** | สแกน `<workspace>/.agents/skills/` แบบ native (มาตรฐาน AgentSkills) | ข้อความธรรมดา |
| **Hermes** | การโหลด skill ตามมาตรฐาน AgentSkills อ่าน `AGENTS.md` | ข้อความธรรมดา |

หกตัวแรกเป็นเอเจนต์เขียนโค้ดแบบโต้ตอบที่มีมนุษย์อยู่ใน session OpenClaw และ Hermes เป็น**แพลตฟอร์มเอเจนต์อัตโนมัติ** — daemon ที่มีอายุยาวนานพร้อมรอบตามกำหนดเวลา — และโดยทั่วไปดำเนินแผนภายใต้โปรไฟล์ไม่มีผู้ดูแล (ดู [โปรไฟล์การดำเนินงาน](#execution-profiles)) ภายใน agent workspace (ดู [Archetype](/spec/archetypes) §3)

เอเจนต์ที่รองรับทุกตัว MUST ถือว่า `AGENTS.md` เป็นแหล่งความจริงเดียวสำหรับข้อตกลงของ repository ไฟล์ config ต่อเอเจนต์ MUST อ้างอิงมันและ MUST NOT ทำซ้ำเนื้อหา

## การออนบอร์ด

ก่อนสร้างหรือดำเนินแผน เอเจนต์ MUST ออนบอร์ดเข้ากับ repository การออนบอร์ดอิงการให้เหตุผล ไม่ใช่อิงสคริปต์ เอเจนต์อ่านโครงสร้าง เอกสาร และการกำหนดค่าของ repository เพื่อสร้างแบบจำลองความคิด

เอเจนต์ SHOULD ระบุ

- archetype ของ repository (repository เดี่ยว orchestrator hub หรือ agent workspace)
- คำสั่ง build, test และ lint
- ข้อตกลงที่มีอยู่สำหรับสไตล์ โครงสร้าง และการตั้งชื่อ
- skill และ agent ที่มีอยู่

Toolchain การทดสอบและการตรวจสอบเป็นบริบทที่จำเป็น ไม่ใช่บริบทที่จะมีหรือไม่ก็ได้ validation gate คือกระดูกสันหลังของแผนที่เชื่อถือได้ ในกรณีที่ repository ตรวจสอบโค้ดอยู่แล้ว เอเจนต์ MUST บันทึกคำสั่งและข้อตกลงของการทดสอบ, lint และ type-check ที่มีอยู่จริงของมัน ในกรณีที่ repository **ไม่มี** toolchain การทดสอบหรือ lint เอเจนต์ MUST NOT เพียงแค่บันทึกว่ามันไม่มีอยู่ — มัน MUST เสนอ toolchain ที่เหมาะกับสแตก (เฟรมเวิร์กและตัวรัน ข้อตกลงไฟล์ทดสอบ เป้าหมายความครอบคลุมเริ่มต้นที่สมเหตุสมผล และเครื่องมือ lint, type-check และ format) บันทึกมันเป็นเป้าหมายในคู่มือการทดสอบ และนำเสนอให้แก่นักพัฒนา repository ที่ไม่มีวิธีที่นิยามไว้สำหรับตรวจสอบพฤติกรรมของมันยังไม่เป็น AI-first

## การวางแผน

เมื่อสร้างแผน เอเจนต์ MUST

- แยกย่อยเป้าหมายเป็นงานตามลำดับที่ทบทวนได้
- เขียนแต่ละงานด้วยโครงสร้างเก้าส่วน
- จบด้วยสามงานสุดท้ายที่บังคับ (การทบทวนความปลอดภัย การค้นพบ Skills และ Agents และรายงานสรุปสำหรับผู้บริหาร)
- ถามคำถามเพื่อให้ชัดเจนเมื่อเป้าหมายกำกวม

## การดำเนินงาน

ระหว่างการดำเนินงาน เอเจนต์ MUST

- อ่านแผนทั้งหมดก่อนเริ่ม
- ดำเนินงานตามลำดับ เว้นแต่ส่วนพึ่งพิงอนุญาตเป็นอื่น
- อัปเดต `PROGRESS.md` หลังแต่ละงาน
- ทำเครื่องหมายสถานะงานอย่างแม่นยำ
- สำหรับงานใด ๆ ที่เพิ่มฟังก์ชันใหม่หรือเปลี่ยนพฤติกรรม ให้เพิ่มหรืออัปเดตการทดสอบอัตโนมัติสำหรับพฤติกรรมนั้นและรันการทดสอบกับการตรวจ lint/type-check ของ repository ก่อนทำเครื่องหมายว่างานเสร็จ อย่าลบหรือข้ามการทดสอบเพื่อบังคับให้ gate ผ่าน
- สำหรับงานใด ๆ ที่แตะการยืนยันตัวตน การจัดการอินพุต ความลับหรือการกำหนดค่า พื้นผิวเครือข่าย หรือ dependency ให้ทำตามความคาดหวังด้านความปลอดภัยที่ประกาศไว้ในเกณฑ์การยอมรับของมัน และยืนยันว่า diff ไม่มีเนื้อหาที่เป็นความลับก่อน commit
- หยุดและถามเมื่อติดขัดแทนที่จะเดา

## การปรับแต่ง

เมื่อปรับแต่ง เอเจนต์ MUST รักษางานที่เสร็จแล้วไว้ อัปเดตตารางงาน และบันทึกสิ่งที่เปลี่ยนแปลง

## การดำเนินต่อ

เมื่อดำเนินต่อ เอเจนต์ MUST ปฏิบัติตาม DWP Resume Protocol ที่นิยามไว้ใน [ข้อกำหนด DWP](/spec/dwp-specification#the-dwp-resume-protocol): re-anchor ไปยัง README ของแผน ระบุ checkpoint ปรับประสาน `state.json` กับ markdown ตรวจสอบรอยต่อ รัน smoke test แล้วดำเนินต่อด้วยงานถัดไปพอดี

## การสื่อสาร

เอเจนต์ SHOULD รายงานอย่างกระชับ รายงานสถานะ MUST แยกแยะงานที่เสร็จแล้ว งานที่กำลังทำ และงานที่ค้างอยู่

## ความปลอดภัย

เอเจนต์ MUST NOT commit ความลับ MUST คง `.dwp/` ไว้ใน gitignore และ SHOULD ถามก่อนทำปฏิบัติการที่ทำลายล้าง การออนบอร์ด MUST ไม่ทำลายข้อมูล เอเจนต์ MUST ตรวจจับไฟล์ที่มีอยู่แล้วและประสานงานแทนที่จะเขียนทับ และ MUST ได้รับการอนุมัติอย่างชัดเจนก่อนแทนที่หรือลบสิ่งใดที่ผู้ใช้มีอยู่แล้ว

วิธีการนี้ยึด Markdown เป็นหลัก ไม่มีการเรียกใช้เครือข่ายและไม่ส่งข้อมูลการวัดระยะไกล เอเจนต์ MUST NOT ส่งออกซอร์สโค้ดหรือความลับออกไปภายนอก ก่อนติดตั้ง skill เอเจนต์ SHOULD ปฏิบัติต่อเนื้อหาการออนบอร์ดที่ดึงมาว่าเป็นอินพุตที่ไม่น่าเชื่อถือ ยืนยันแหล่งที่มาจากแหล่งข้อมูลทางการ และตรวจสอบการเผยแพร่กับ checksum ที่เผยแพร่ไว้

## โปรไฟล์การดำเนินงาน

ทุกแผนดำเนินงานภายใต้โปรไฟล์หนึ่งในสองอย่างพอดี โปรไฟล์เปลี่ยนผู้เฝ้าดู ไม่ใช่ gate ที่ใช้ — วินัยการตรวจสอบเหมือนกันในทั้งสอง

### แบบโต้ตอบ (ค่าเริ่มต้น)

มนุษย์อยู่ใน session เอเจนต์เสนอ มนุษย์อนุมัติร่างที่ปรับแต่งแล้ว เอเจนต์ดำเนินงานทีละงาน และความคลุมเครือถูกแก้ไขโดยการถาม ส่วนโปรโตคอลทั้งหมดข้างต้นอธิบายโปรไฟล์แบบโต้ตอบ

### แบบไม่มีผู้ดูแล

แผนรันโดยไม่มีมนุษย์เฝ้าดู — รอบตามกำหนดเวลาของแพลตฟอร์มอัตโนมัติ cloud session การรันข้ามคืน การดำเนินงานแบบไม่มีผู้ดูแลเป็นแบบเลือกเข้าร่วมต่อแผนและ MUST เป็นไปตามทั้งหมดต่อไปนี้

- **แผนที่อนุมัติล่วงหน้า** ร่างที่ปรับแต่งแล้วได้รับการอนุมัติจากมนุษย์ก่อนรอบไม่มีผู้ดูแลใด ๆ เอเจนต์ MUST NOT สร้างและดำเนินแผนแบบไม่มีผู้ดูแลในรอบเดียว การอนุมัติแผนคือจุดควบคุมของมนุษย์
- **ชั้นสถานะ REQUIRED** แผน MUST พก `manifest.json` และ `state.json` เพื่อให้ session ในภายหลัง — เอเจนต์หรือมนุษย์ — สามารถอ่านความคืบหน้าที่แม่นยำโดยไม่ต้องเล่น transcript ซ้ำ ดู [สถานะแผน](/spec/plan-state)
- **อำนาจจำกัด** อำนาจของเอเจนต์คือแผน: มัน MUST NOT ขยายขอบเขต MUST NOT ดำเนินการทำลายหรือส่งออกภายนอกที่แผนไม่ได้อนุญาตอย่างชัดเจน และ MUST NOT ยืดคำสั่งของงานเพื่อครอบคลุมงานที่ค้นพบแต่ไม่ได้วางแผน — งานที่ค้นพบถูกบันทึกไว้สำหรับ `refine` ถัดไป ไม่ใช่ด้นสด
- **หนึ่งงาน atomic ต่อรอบ gate เสมอ** แต่ละรอบรัน DWP Resume Protocol ดำเนินอย่างมากงานถัดไป ผ่าน validation gate ของมัน เสร็จสิ้นตามโปรโตคอลการเสร็จสิ้นงาน และ yield gate ที่ล้มเหลวคือเงื่อนไขหยุด ไม่ใช่ "ดำเนินต่อไปได้"

### เงื่อนไขหยุดและการส่งต่อเพื่อแก้ไข

เอเจนต์ไม่มีผู้ดูแล MUST หยุดแผน — เติมฟิลด์ `blocked` ของ `state.json` ด้วยงาน เหตุผล และสิ่งที่ต้องการ แล้วหยุด — เมื่อเกิดเหตุการณ์ใดต่อไปนี้:

1. validation gate ล้มเหลวและการแก้ไขไม่อยู่ในขอบเขตของงานแล้ว
2. งานต้องการการอนุมัติ ข้อมูลรับรอง หรือการตัดสินใจที่แผนไม่ได้อนุมัติล่วงหน้า
3. ความเป็นจริงแตกต่างจากสมมติฐานของแผน (ไฟล์หายไป API เปลี่ยน งานพร้อมกันที่ขัดแย้ง หรือการเบี่ยงเบนที่การปรับประสานไม่สามารถแก้ไขได้)
4. สองรอบติดต่อกันไม่มีความคืบหน้าที่ตรวจสอบได้บนงานเดียวกัน

การหยุดคือความสำเร็จ ไม่ใช่ความล้มเหลว บันทึก blocked คือข้อความส่งต่อเพื่อแก้ไข ช่องทางการแจ้งเตือนของแพลตฟอร์ม SHOULD แสดงมัน มนุษย์ (หรือ session `refine`) ปลดล็อก และรอบตามกำหนดเวลาถัดไปดำเนินต่อตามปกติ

### Scheduled continuation

บนแพลตฟอร์มที่มีการกำหนดเวลา — heartbeat หรือ cron ของ OpenClaw, cron ของ Hermes, cloud-agent wake — การดำเนินต่อ MUST แสดงเป็น: *wake → รัน DWP Resume Protocol → ถ้า `blocked` รายงานและ yield → มิฉะนั้นดำเนินงาน atomic ถัดไป → อัปเดตชั้นสถานะ → yield* แผน ไม่ใช่เซสชัน คือหน่วยของความต่อเนื่อง แผน MUST รอดจากการรีสตาร์ตของแพลตฟอร์ม การเปลี่ยนโมเดล หรือเอเจนต์อื่นที่รับรอบถัดไป
