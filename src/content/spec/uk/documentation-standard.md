# Стандарт документації

**Версія 1.0.** Цей стандарт визначає, як Deep Work Plan документують свою структуру, завдання та прогрес. Він застосовується до кожного плану, створеного за методологією DWP. Ключові слова MUST, SHOULD та MAY вживаються згідно з RFC 2119.

## README плану

Кожен план MUST мати `README.md`, що містить:

- **Заголовок** — `# Deep Work Plan: <name>`.
- **Мета** — прозовий виклад завдання плану.
- **Вихідний матеріал** — посилання чи шляхи до канонічних вхідних даних (необовʼязково).
- **Завдання** — markdown-таблиця з номером завдання, назвою та чекбоксом статусу.
- **Статус** — рядок у формі `<n>/<total> tasks complete`.

## Файли завдань

Кожен файл завдання MUST мати назву `<n>.task_<slug>.md` і містити девʼятисекційну анатомію.

## PROGRESS.md

`PROGRESS.md` — це журнал виконання лише для додавання. Кожен запис MUST фіксувати:

- Позначку часу ISO 8601.
- Номер і назву завдання.
- Що було зроблено.
- Будь-які відхилення чи причини пропуску.

## Маркери статусу

- `[ ]` — не розпочато.
- `[~]` — у процесі.
- `[x]` — завершено.
- `[!]` — заблоковано.

## Заголовки

Усі заголовки MUST використовувати регістр речення. Документи SHOULD уникати маркетингової мови та знаків оклику.

## Три обовʼязкові фінальні завдання

Кожен план MUST закінчуватися трьома стандартними завданнями:

1. **Security Review** — провести аудит повного набору змін плану на секрети, ризики інʼєкцій та нову поверхню атаки й перевірити, що `docs/SECURITY.md` досі відображає реальність. Критична знахідка блокує завершення.
2. **Skills & Agents Discovery** — виявити створені багаторазові скіли й агенти.
3. **Executive Report** — підсумок результатів для зацікавлених сторін.
