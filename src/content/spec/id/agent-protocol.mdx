# Agent protocol

**Versi 1.2.** Protokol ini mendefinisikan bagaimana sebuah AI coding agent MUST berperilaku saat bekerja dengan Deep Work Plan. Kata kunci MUST, SHOULD, dan MAY mengikuti RFC 2119.

> **Tambahan di v1.2.** Dua tambahan, tanpa perubahan yang merusak: (1) platform agent otonom (OpenClaw, Hermes) bergabung dalam tabel supported agents; (2) bagian **Profil eksekusi** mendefinisikan eksekusi tanpa pengawasan — otoritas terbatas, lapisan status wajib, kondisi berhenti, dan kelanjutan terjadwal.

## Supported agents

Metodologi ini MUST mendukung AI coding agent berikut. Agent masa depan mana pun yang membaca markdown dan dapat mengeksekusi tool call MAY ditambahkan tanpa perubahan yang merusak.

| Agent | Konvensi konfigurasi native | Prefix command |
|-------|--------------------------|----------------|
| **Claude Code** | `.claude/` (symlink ke `.agents/`) | `/` (slash command native) |
| **Cursor** | `.cursor/rules/*.mdc` yang merujuk `AGENTS.md` | `#` atau plain text |
| **OpenAI Codex** | `.codex/` yang merujuk `AGENTS.md` | `#` atau plain text |
| **Google Gemini** | `.gemini/` yang merujuk `AGENTS.md` | `#` atau plain text |
| **GitHub Copilot** | `.github/copilot-instructions.md` yang merujuk `AGENTS.md` | `#` atau plain text |
| **Antigravity** | `.antigravity/` yang merujuk `AGENTS.md` | `#` atau plain text |
| **OpenClaw** | Secara native memindai `<workspace>/.agents/skills/` (standar AgentSkills) | plain text |
| **Hermes** | Pemuatan skill standar AgentSkills; membaca `AGENTS.md` | plain text |

Enam yang pertama adalah interactive coding agent dengan manusia dalam sesi. OpenClaw dan Hermes adalah **platform agent otonom** — daemon long-lived dengan giliran terjadwal — dan biasanya mengeksekusi rencana di bawah profil tanpa pengawasan (lihat [Profil eksekusi](#execution-profiles)) di dalam ruang kerja agent (lihat [Arketipe](/spec/archetypes) §3).

Setiap supported agent MUST memperlakukan `AGENTS.md` sebagai satu-satunya sumber kebenaran untuk konvensi repositori. Berkas konfigurasi per-agent MUST merujuknya dan MUST NOT menduplikasi isinya.

## Onboarding

Sebelum membuat atau mengeksekusi sebuah rencana, sebuah agent MUST meng-onboard ke repositori. Onboarding bersifat berbasis penalaran, bukan berbasis skrip: agent membaca struktur, dokumentasi, dan konfigurasi repositori untuk membangun model mental.

Agent SHOULD mengidentifikasi:

- Arketipe repositori (repositori individual, orchestrator hub, atau ruang kerja agent).
- Perintah build, test, dan lint.
- Konvensi yang ada untuk gaya, struktur, dan penamaan.
- Skill dan agent yang tersedia.

Toolchain pengujian dan validasi adalah konteks yang esensial, bukan opsional: validation gate adalah tulang punggung dari rencana yang andal. Di mana repositori sudah memvalidasi kode, agent MUST mencatat perintah dan konvensi test, lint, dan type-check-nya yang nyata. Di mana repositori **tidak** memiliki toolchain test atau lint, agent MUST NOT sekadar mencatat ketiadaannya — ia MUST mengusulkan satu yang sesuai dengan stack (sebuah framework dan runner, sebuah konvensi berkas-test, sebuah target cakupan awal yang masuk akal, serta perkakas lint, type-check, dan format), mendokumentasikannya sebagai target di panduan pengujian, dan mengangkatnya ke pengembang. Sebuah repositori tanpa cara yang terdefinisi untuk memvalidasi perilakunya belum AI-first.

## Perencanaan

Saat membuat sebuah rencana, agent MUST:

- Menguraikan tujuan menjadi tugas-tugas berurutan yang dapat ditinjau.
- Menulis setiap tugas dengan anatomi sembilan bagian.
- Mengakhiri dengan dua tugas akhir wajib.
- Mengajukan pertanyaan klarifikasi ketika tujuannya ambigu.

## Eksekusi

Selama eksekusi, agent MUST:

- Membaca seluruh rencana sebelum memulai.
- Mengeksekusi tugas secara berurutan kecuali dependensi mengizinkan sebaliknya.
- Memperbarui `PROGRESS.md` setelah setiap tugas.
- Menandai status tugas secara akurat.
- Untuk setiap tugas yang menambahkan fungsionalitas baru atau mengubah perilaku, menambahkan atau memperbarui test otomatis untuk perilaku itu dan menjalankan test repositori serta pemeriksaan lint/type-check sebelum menandai tugas selesai; tidak pernah menghapus atau melewati sebuah test untuk memaksa gate lulus.
- Berhenti dan bertanya saat terhambat alih-alih menebak.

## Penyempurnaan

Saat menyempurnakan, agent MUST mempertahankan pekerjaan yang selesai, memperbarui tabel tugas, dan mencatat apa yang berubah.

## Pelanjutan

Saat melanjutkan, agent MUST mengikuti Protokol Resume DWP yang didefinisikan di [Spesifikasi DWP](/spec/dwp-specification#the-dwp-resume-protocol): re-anchor ke README rencana, temukan checkpoint, rekonsiliasi `state.json` terhadap markdown, periksa seam, jalankan smoke test, kemudian lanjutkan dengan tepat tugas berikutnya.

## Komunikasi

Agent SHOULD melapor secara padat. Laporan status MUST membedakan pekerjaan yang selesai, sedang berjalan, dan tertunda.

## Keamanan

Agent MUST NOT meng-commit secret, MUST menjaga `.dwp/` tetap di-gitignore, dan SHOULD bertanya sebelum operasi yang destruktif. Onboarding MUST bersifat non-destruktif: sebuah agent MUST mendeteksi file yang sudah ada dan merekonsiliasi alih-alih menimpa, serta MUST memperoleh persetujuan eksplisit sebelum mengganti atau menghapus apa pun yang sudah dimiliki pengguna.

Metodologi ini bersifat Markdown-first: tidak melakukan panggilan jaringan apa pun dan tidak memancarkan telemetri, serta sebuah agent MUST NOT mengeksfiltrasi sumber kode atau secret. Sebelum menginstal skill, sebuah agent SHOULD memperlakukan konten onboarding yang diambil sebagai input yang tidak tepercaya, mengonfirmasi asal-usulnya dari sumber resmi, dan memverifikasi rilis tersebut terhadap checksum yang dipublikasikan.

## Profil eksekusi

Setiap rencana dieksekusi di bawah tepat satu dari dua profil. Profil mengubah siapa yang mengawasi, tidak pernah gate apa yang berlaku — disiplin validasi identik di keduanya.

### Interaktif (default)

Seorang manusia hadir dalam sesi. Agent mengusulkan, manusia menyetujui draf yang telah disempurnakan, agent mengeksekusi tugas demi tugas, dan ambiguitas diselesaikan dengan bertanya. Semua bagian protokol di atas menggambarkan profil interaktif.

### Tanpa pengawasan

Rencana berjalan tanpa manusia yang mengawasi — giliran terjadwal platform otonom, sesi cloud, jalannya semalam. Eksekusi tanpa pengawasan bersifat opt-in per rencana dan MUST memenuhi semua hal berikut:

- **Rencana yang telah disetujui sebelumnya.** Draf yang disempurnakan telah disetujui oleh manusia sebelum giliran tanpa pengawasan mana pun. Sebuah agent MUST NOT membuat dan mengeksekusi rencana tanpa pengawasan dalam satu giliran; persetujuan rencana adalah titik kontrol manusia.
- **Lapisan status REQUIRED.** Rencana MUST membawa `manifest.json` dan `state.json` sehingga sesi berikutnya — agent atau manusia — dapat membaca kemajuan yang tepat tanpa memutar ulang transkrip. Lihat [Status rencana](/spec/plan-state).
- **Otoritas terbatas.** Otoritas agent adalah rencana: ia MUST NOT memperluas cakupan, MUST NOT melakukan tindakan destruktif atau menghadap keluar yang tidak secara eksplisit diotorisasi rencana, dan MUST NOT meregangkan instruksi tugas untuk mencakup pekerjaan yang ditemukan-tetapi-tidak-direncanakan — pekerjaan yang ditemukan dicatat untuk `refine` berikutnya, tidak diimprovisasi.
- **Satu tugas atomik per giliran, gate selalu.** Setiap giliran menjalankan Protokol Resume DWP, mengeksekusi paling banyak tugas berikutnya, melewati validation gate-nya, menyelesaikan per protokol penyelesaian tugas, dan yield. Gate yang gagal adalah kondisi berhenti, tidak pernah "lanjutkan saja".

### Kondisi berhenti dan eskalasi

Sebuah agent tanpa pengawasan MUST menghentikan rencana — mengisi kolom `blocked` di `state.json` dengan tugas, alasan, dan apa yang dibutuhkan, kemudian berhenti — ketika salah satu dari hal ini terjadi:

1. Sebuah validation gate gagal dan perbaikannya tidak berada dalam cakupan tugas.
2. Tugas memerlukan persetujuan, kredensial, atau keputusan yang tidak diotorisasi rencana sebelumnya.
3. Realita menyimpang dari asumsi rencana (berkas hilang, API berubah, pekerjaan konkuren yang bertentangan, atau desinkronisasi yang tidak dapat diselesaikan rekonsiliasi).
4. Dua giliran berturut-turut tidak menghasilkan kemajuan yang dapat diverifikasi pada tugas yang sama.

Berhenti adalah keberhasilan, bukan kegagalan: catatan terblokir adalah pesan eskalasi. Saluran notifikasi platform SHOULD menampilkannya; manusia (atau sesi `refine`) membuka blokiran, dan giliran terjadwal berikutnya melanjutkan secara normal.

### Kelanjutan terjadwal

Pada platform dengan penjadwalan — heartbeat atau cron OpenClaw, cron Hermes, bangun cloud-agent — kelanjutan MUST dinyatakan sebagai: *bangun → jalankan Protokol Resume DWP → jika `blocked`, laporkan dan yield → jika tidak, eksekusi tugas atomik berikutnya → perbarui lapisan status → yield.* Rencana, bukan sesi, adalah unit kontinuitas; sebuah rencana MUST bertahan dari platform yang restart, model yang berubah, atau agent yang berbeda yang mengambil alih giliran berikutnya.
