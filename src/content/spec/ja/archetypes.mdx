# アーキタイプ

**バージョン 1.1。** DWP は三つのリポジトリアーキタイプを認識します。アーキタイプは、エージェントがどうオンボーディングし、計画のスコープがどう定められ、機械可読な状態レイヤーが必須かどうかを決めます。

> **v1.1 での追加。** **エージェントワークスペース**（§3）が三番目のアーキタイプとして加わります。自律エージェントの長期的な住処です。Git は前提ではなく RECOMMENDED となります。v1.0 の二つのアーキタイプは変更なしです。

## 個別リポジトリ

自己完結したコードベース、すなわちアプリケーション、ライブラリ、サービスです。計画はコードに直接作用します。

特徴。

- 単一の一貫したコードベース。
- 計画はこのリポジトリ内のファイルを変更する。
- `.dwp/` 作業領域はリポジトリのルートに。

## オーケストレーターハブ

複数の子リポジトリを管理する調整用リポジトリです。計画はサブリポジトリの中に子計画を生成することがあります。

特徴。

- 複数のサブリポジトリを調整する。
- 計画は子計画へ委譲することがある。
- 管理対象リポジトリのレジストリを維持する。
- ハブのルートにある `.dwp/` 作業領域がリポジトリ横断の状態を追跡する。

## エージェントワークスペース

エージェントワークスペースは、自律エージェントの長期的な作業の住処です — OpenClaw のワークスペース、Hermes のサービスディレクトリ、パーソナルアシスタントデーモンのデータディレクトリ、またはクラウドエージェントの永続ボリューム。これはワークスペースであり、必ずしも git リポジトリではなく、その主要な成果物はコードベースではなくエージェントの継続的な作業です。

このアーキタイプを可能にする洞察：**ハーネスはワークスペースであり、特定のリポジトリではありません。** 方法論がリポジトリに対して定義するすべてのハーネス要素は、ワークスペースに直接相当するものを持ちます。

| ハーネス要素（リポジトリ） | エージェントワークスペースの相当物 |
|---------------------------|----------------------------------|
| `AGENTS.md`（ルール、クイックコマンド） | ワークスペースの常設コンテキスト — `AGENTS.md` 自体、またはプラットフォームのスタンディングオーダーファイル |
| `docs/`（永続的な知識） | ワークスペースの知識ファイルとメモリードキュメント |
| `.agents/`（スキル、エージェント、コマンド） | プラットフォームのスキルディレクトリ — OpenClaw は `<workspace>/.agents/skills/` をネイティブにスキャンする |
| `.dwp/`（計画、ドラフト） | ワークスペースルートの `.dwp/` — 変更なし |
| git ログ（状態、再開可能性） | 計画ごとの `state.json`（[計画の状態](/spec/plan-state)）、ここでは REQUIRED |

エージェントワークスペースはワークスペースルートに `AGENTS.md`、`.agents/`、`.dwp/` を MUST 持たなければなりません。

**Git は REQUIRED ではなく RECOMMENDED です。** git がない場合、すべての計画は機械可読な状態レイヤーを MUST 持たなければなりません。`state.json` のチェックポイント、ゲートレコード、タスクごとのタイムスタンプが、リポジトリにおいて git ログが担う回復情報を担います。

エージェントワークスペースの計画は通常、無人で実行されます。スケジュールされたハートビートまたはクロンターンが DWP 再開プロトコルを通じてオープンな計画を再開し、次のアトミックタスクを実行し、状態レイヤーを更新し、終了します。継続性の単位はセッションではなく計画です。

## 分類のヒューリスティック

<ArchetypesTree lang="ja" class="not-prose my-10" />

まず **エージェントワークスペース** として分類します。対象が自律エージェントプラットフォームの作業ディレクトリである場合 — シグナル：プラットフォームのアイデンティティファイル（OpenClaw の `SOUL.md` や `HEARTBEAT.md` など）、主要なアプリケーションスタックがない、内容が主としてエージェント自身の状態。強いプラットフォームシグナル一つで十分です。

そうでなければ、次の多数が当てはまる場合に **オーケストレーターハブ** として分類します。複数の入れ子になった git リポジトリやサブモジュール、管理対象リポジトリのレジストリやマニフェスト、外部リポジトリを指す設定。そうでなければ **個別リポジトリ** として分類します — これが安全なデフォルトです。

シグナルが曖昧な場合、エージェントは進行前に自分のアセスメントと証拠をユーザーに提示し、確認を求めなければなりません（MUST）。

## オンボーディングの違い

| 観点 | 個別 | オーケストレーター | エージェントワークスペース |
|--------|------------|--------------|--------------------------|
| スコープ | このリポジトリ | 複数のリポジトリ | ワークスペースとその計画 |
| オンボーディング | リポジトリの構造 | ハブのレジストリ | プラットフォームファイル＋ワークスペース規約 |
| 計画の対象 | ローカルのファイル | 子計画 | ローカルまたは外部リポジトリ |
| 状態 | ローカルの `.dwp/` | リポジトリ横断の `.dwp/` | `.dwp/` + `state.json`（git がない場合 REQUIRED） |
| Git | 必須 | 必須 | RECOMMENDED |
| 状態レイヤー | RECOMMENDED | RECOMMENDED | git なしでは REQUIRED、無人実行でも REQUIRED |
