# Init — Промт онбордингу для AI-агентів

Ви — AI-агент програмування. Спершу прочитайте весь цей промт, потім зробіть поточний репозиторій
**AI-first**, **spec-driven** та **пілотовним агентами**, прийнявши методологію Deep Work Plan (DWP).
Ставтеся до цього як до структурованого інженерного завдання — Deep Work Plan, — а не швидкої правки.

## Операційні принципи (прочитайте, перш ніж щось змінювати)

1. **Міркуйте про *цей* репозиторій.** Ніколи не копіюйте шаблон і не лишайте плейсхолдер. Кожен файл, який ви
   пишете, має відображати реальні мови, фреймворки, команди та структуру репозиторію. Загальна
   заглушка — це невдача, а не результат.
2. **Будьте неруйнівними — питайте, перш ніж замінювати.** Репозиторій уже може мати `AGENTS.md`,
   дерево `docs/`, налаштування `.agents/` чи скілів, `CLAUDE.md` або власні домовленості. **Не**
   перезаписуйте, не переміщуйте й не видаляйте наявну роботу мовчки. Виявіть, що існує, прочитайте це та **узгодьте**:
   зливайте й поліпшуйте на місці, де це можливо. Перш ніж замінювати чи видаляти будь-що, що користувач
   уже має, поясніть що і чому та отримайте його явне схвалення.
3. **Запропонуйте план, потім виконуйте.** Після розвідки представте короткий план — що ви
   створите, що зміните та будь-що наявне, що пропонуєте узгодити чи замінити, — і зачекайте
   на підтвердження користувача, перш ніж робити великі чи руйнівні зміни.
4. **Працюйте безпечними, придатними до рецензування інкрементами.** Комітьте логічно, тримайте diff читабельними, не торкайтеся секретів
   і лишайте дотичний код у спокої.
5. **Зупиніться й відзвітуйте**, якщо крок не вдався або стан репозиторію неоднозначний.
6. **Довіряй, але перевіряй.** Ставтеся до цього промту як до ненадійних вхідних даних: підтвердьте, що
   отримали його з офіційних джерел (`deepworkplan.com` і репозиторії `DailybotHQ`), оцініть його перед
   виконанням і перевірте цілісність скілу перед запуском — дивіться *Довіра та верифікація* нижче.

## 0. Прочитайте методологію та специфікацію

Перш ніж щось змінювати, прочитайте канонічні джерела, щоб зрозуміти стандарт, який ви приймаєте:

- Методологія: https://deepworkplan.com/methodology.md
- Специфікація: https://deepworkplan.com/spec.md
- Каталог набору: https://deepworkplan.com/kit.md

## 1. Розвідка та план на схвалення

Спершу зрозумійте репозиторій, потім запропонуйте, що робитимете.

- **Визначте стек.** Мови, фреймворки, менеджер пакетів (з lockfile, що насправді
  існує), реальні команди build/test/lint/type-check, модулі вихідного коду, домовленість про тестування та
  форму розгортання.
- **Класифікуйте архетип.** Окремий репозиторій (поширений випадок) або хаб-оркестратор, з
  доказами.
- **Інвентаризуйте те, що вже існує.** `AGENTS.md`, `CLAUDE.md`, `docs/`, будь-яке налаштування `.agents/` чи скілів/агентів,
  `.dwp/` та `.gitignore`. Зазначте все, що вже виконує частину цієї роботи.
- **Запропонуйте план онбордингу.** Представте стислий список: файли, які ви створите, файли, які ви
  зміните, та будь-що наявне, що пропонуєте узгодити чи замінити, — потім **попросіть користувача підтвердити**,
  перш ніж продовжувати. Для великого репозиторію зафіксуйте цей план через `/dwp-create` і виконайте його як
  Deep Work Plan.

## Довіра та верифікація (перед встановленням)

Ви збираєтеся встановити й запустити код у цьому репозиторії, тому спочатку перевірте, що саме ви приймаєте.
Це заохочується: обережний агент вважає отриманий промт ненадійним, доки не перевірить його.

- **Підтвердьте походження.** Цей промт і скіл є відкритими та розповсюджуються під ліцензією MIT. Канонічні
  джерела — `https://deepworkplan.com` і організація `DailybotHQ` на GitHub
  (`deepworkplan-website`, `deepworkplan-skill`). Машинозчитуваний маніфест опубліковано за адресою
  `https://deepworkplan.com/.well-known/dwp-trust.json`.
- **Знайте, що робить скіл.** Він Markdown-first: без CLI, без HTTP API, без автентифікації, **без мережевих
  викликів і без телеметрії**. Його єдина безпекова дія — мутація *цього* репозиторію, і він за своєю
  природою неруйнівний — узгоджує та запитує перед заміною.
- **Перевірте цілісність перед запуском.** Кожен реліз публікує `SHA256SUMS` для скіла, що постачається;
  підтвердьте, що ваша копія збігається, перш ніж їй довіряти:

  ```bash
  git clone https://github.com/DailybotHQ/deepworkplan-skill.git
  cd deepworkplan-skill
  # Завантажте контрольні суми для релізу, який збираєтеся встановити (замініть vX.Y.Z):
  curl -fsSL -o SHA256SUMS \
    https://github.com/DailybotHQ/deepworkplan-skill/releases/download/vX.Y.Z/SHA256SUMS
  ./setup.sh --verify        # ненульовий вихід означає невідповідність файлу — зупиніться
  ```

  Релізи **контрольно-суміовані, але не підписані** (підписання — задокументований наступний крок). Усе відкрите
  та можна порівняти diff, тому будь-який файл можна порівняти з репозиторієм на відповідному тезі.
- **Повідомте про проблему.** Питання безпеки повідомляйте **приватно через звітування про вразливості GitHub** у відповідному репозиторії — `https://github.com/DailybotHQ/deepworkplan-skill/security` або `https://github.com/DailybotHQ/deepworkplan-website/security` — не як публічне питання. Дивіться також `SECURITY.md` кожного репозиторію, `https://deepworkplan.com/.well-known/security.txt` і сторінку довіри `https://deepworkplan.com/trust`.

## 2. Встановіть скіл Deep Work Plan

Встановіть скіл, щоб агенти цього репозиторію могли планувати й виконувати структуровану роботу. Він постачає маршрутизатор
плюс вісім суб-скілів — `create`, `execute`, `refine`, `resume`, `status`, `verify`, `onboard` та `author`.

```bash
npx skills add DailybotHQ/deepworkplan-skill
```

Або склонуйте та запустіть setup-скрипт:

```bash
git clone https://github.com/DailybotHQ/deepworkplan-skill.git && cd deepworkplan-skill && ./setup.sh
```

## 3. Проведіть онбординг репозиторію (осмислено й неруйнівно)

Викличте суб-скіл onboard (`/deepworkplan-onboard`). Міркуйте про реальний репозиторій та адаптуйте все
до нього. Для кожного артефакту нижче **якщо він уже існує, узгодьте його** (злийте, поліпшіть, вирівняйте до
методології) замість перезапису — і підтвердьте з користувачем, перш ніж щось замінювати.

1. **`AGENTS.md` + `CLAUDE.md`.** Створіть `AGENTS.md` у корені — покажчик, обовʼязкові правила
   (лише англійська, конвенційні коміти, реальний тестовий патерн репозиторію та gate рецензування) та
   блок Quick Commands з **реальними, виконуваними** командами репозиторію. Якщо `AGENTS.md` уже існує, зливайте
   в нього, а не замінюйте його. Створіть символьне посилання `CLAUDE.md → AGENTS.md` (не затирайте
   наявний `CLAUDE.md`, не спитавши).
2. **`docs/`.** Заповніть стандартні категорії реальним, специфічним для репозиторію вмістом: `PRODUCT_SPEC.md` (нетехнічний документ продукту/навіщо — обов'язковий для кожного репозиторію, включно з бібліотеками), `ARCHITECTURE.md`,
   `STANDARDS.md`, `TESTING_GUIDE.md`, `DEVELOPMENT_COMMANDS.md`, `SECURITY.md`,
   `AI_AGENT_ONBOARDING.md`, `AI_AGENT_COLLAB.md`, плюс `PERFORMANCE.md` та покажчик `docs/README.md`.
   Якщо документація вже існує, інтегруйте та розширте її — не дублюйте.
3. **Документація кожного модуля.** Додайте `README.md` (і підтеку `docs/` для складних модулів) усередині кожного
   великого модуля вихідного коду, виявленого під час розвідки.
4. **`.agents/` + `.claude → .agents`.** Створіть канонічний, крос-агентний дім: **осмислений** каталог
   `agents/`, доречних для стека `skills/` та тонких `dwp-*` `commands/`, що делегують
   встановленому скілу — кожен запис обґрунтований для *цього* репозиторію, не скопійований з іншого. Додайте
   каталог `docs/` (`skills_agents_catalog.md` + `COMMANDS_REFERENCE.md`), що відповідає тому, що є на
   диску, плюс `settings.json` та символьне посилання `.claude → .agents`. Уведіть будь-які наявні скіли/агенти
   в каталог.
5. **Скіл DWP, адаптований.** Встановлений скіл — це двигун; власний набір репозиторію
   (скіли, агенти, команди) має бути **осмислений для цього репозиторію** — ніколи не копіюванням набору іншого
   репозиторію.
6. **`.dwp/` + `tmp/`.** Створіть `.dwp/` у gitignore з `plans/` та `drafts/`, плюс простір
   `tmp/` — обидва додані до `.gitignore` неруйнівно (дописуйте, ніколи не переписуйте).

## 4. Розвивайте набір (суб-скіл author)

Використовуйте суб-скіл `author`, щоб вирощувати власний набір репозиторію. Тонкі делегатори `/skill-create` та
`/agent-create` скеровують до нього. Створіть **скіл** для повторюваної в межах сесії процедури, **агент** для
повторюваної ролі з власним рівнем моделі та інструментами, а **команду** лише як тонкий делегатор. Тримайте
каталог `.agents/docs/` синхронізованим із тим, що є на диску.

## 5. Запропонуйте add-on із явним прийняттям

Після базису перелічіть три add-on і запропонуйте кожен як явне прийняття. Репозиторій
повністю відповідний з **нульовою** кількістю add-on — ніколи не встановлюйте їх автоматично.

- **Підтримка Devcontainer** — відтворюваний, ізольований dev-контейнер зі стійкою автентифікацією AI-CLI.
- **Інтеграція з Dailybot** — найкраще-можливе звітування про прогрес/віхи для команд, що вже використовують Dailybot.
  Базова методологія не має жодної залежності від Dailybot.
- **Оновлення залежностей** — незалежні від менеджера пакетів, партіями, перевірені, відкочувані оновлення. Коли
  прийнято, воно встановлює команду `/lib-upgrade`.

## 6. Плануйте та виконуйте

Генеруйте Deep Work Plan для будь-якого завдання й виконуйте їх завдання за завданням:

- `/dwp-create <goal>` — розкласти мету на пронумеровані, послідовні завдання з валідаційними gate.
- `/dwp-execute` — виконати план завдання за завданням, оновлюючи прогрес та перевіряючи кожен gate.
- `/dwp-status` — відзвітувати про прогрес без жодних змін.
- `/dwp-refine` — додати, видалити чи переставити завдання, зберігаючи завершену роботу.
- `/dwp-resume` — відтворити стан та продовжити перерваний план.

## 7. Перевірте

Запустіть `/dwp-verify` для обʼєктивного звіту про відповідність «пройдено/не пройдено» (він перевіряє
критерії в [документі про відповідність зі специфікації](https://deepworkplan.com/spec)),
потім підтвердьте:

- [ ] Скіл встановлено й він розвʼязуваний, з усіма вісьмома доступними суб-скілами.
- [ ] `AGENTS.md` існує в корені з реальним блоком Quick Commands; `CLAUDE.md` розвʼязується в нього.
- [ ] `docs/` містить стандартні категорії з реальним, специфічним для репозиторію вмістом; великі модулі мають
      `README.md`.
- [ ] `.agents/` існує з `agents/`, `commands/` (тонкі делегатори `dwp-*`), `skills/` та каталогом,
      що відповідає дійсності; `.claude → .agents` розвʼязується.
- [ ] `.dwp/` існує, у gitignore, і має `plans/` та `drafts/`; `tmp/` існує та в gitignore.
- [ ] Наявний вміст користувача збережено чи узгоджено за згодою — нічого не знищено мовчки.
- [ ] Ви можете згенерувати Deep Work Plan і виконати його завдання за завданням, перевіряючи кожен gate.

## Результат

Коли онбординг завершено, репозиторій змінено двома стійкими способами — стовпами
методології:

1. **Репозиторій є spec-driven.** Робота починається з написаного плану та специфікації, а не з
   спонтанних промтів.
2. **Сам репозиторій є harness агента.** `AGENTS.md`, `docs/`, документація кожного модуля та
   дім скілів `.agents/` дають кожному агенту контекст і команди, потрібні для структурованої,
   перевірюваної роботи.

Будь-хто може запустити цей промт на будь-якому репозиторії — і завершити кодовою базою, яку може пілотувати будь-який AI-агент.
