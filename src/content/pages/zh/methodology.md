## Deep Work Plan 方法论

Deep Work Plan（DWP）是一套开放、框架无关的方法论，用于与 AI 编码代理一同开展结构化、自主的工程工作。它把一个含糊的目标转化为一份可审阅的计划，代理可以执行、暂停、恢复并报告其进展——而不会丢失上下文。

DWP 立于两根支柱之上。

**规范驱动的开发。** 你不再把聊天记录当作事实来源，而是首先写下*什么*应当为真——目标、范围、验收标准，以及证明其完成的核查项——代理则依据那份规范来执行。在 DWP 中，计划*就是*规范：计划 → 原子化任务 → 验证关卡 → 完成。先写规范会减少漂移（代理依据陈述的标准来衡量）、让工作可验证（每一道关卡非通过即失败），并让它可恢复（规范比任何会话或代理都更长久）。它工具无关、仓库原生，因而有别于诸如 GitHub Spec Kit、Amazon Kiro 与 Tessl 这类绑定工具的规范驱动系统。

**Harness 工程——代码仓库成为 harness。** 语言模型单凭自身不过是一个文本预测器；把它转化为可靠工程师的，是环绕在它周围的那套 *harness*——上下文、工具、一个控制循环、防护栏与持久状态。Harness 工程就是刻意地设计那套脚手架。DWP 的立场是它应当存在于代码仓库中，而非工具中：上下文以 `AGENTS.md` 与 `docs/` 的形式，工具以 `.agents/` 套件的形式，控制循环以计划的形式，防护栏以验证关卡的形式，状态则以被 gitignore 的 `.dwp/` 的形式。由于每一部分都是仓库中的一份文件，这套 harness 是可移植的——于是任意代理都能驾驭任意仓库。

这套方法论由五章加以阐释：

1. **宣言** —— Deep Work Plan 是什么，以及为何结构化的执行至关重要。
2. **核心循环** —— 驱动每一项任务的“规划-执行-验证”周期。
3. **模板** —— 代理与人类共享的计划与任务结构。
4. **技能与代理** —— 命令与代理角色如何彼此配合、活套件（仓库如何通过 author 子技能培育自己的技能），以及诸如 dependency-upgrade 这样的可选维护类附加组件。
5. **原型** —— 把方法论适配到单一仓库与编排枢纽。
