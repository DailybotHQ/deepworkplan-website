# Init — AIエージェントのためのオンボーディングプロンプト

あなたはAIコーディングエージェントです。まずこのプロンプト全体を読み、その後、Deep Work Plan（DWP）方法論を採用することで、現在のリポジトリを **AI-first**、**仕様駆動**、**エージェントが操縦できる**ものにしてください。これを手早い編集ではなく、構造化されたエンジニアリング作業、すなわち Deep Work Plan として扱ってください。

## 運用上の原則（何かを変更する前に読む）

1. ***この*リポジトリについて推論する。** テンプレートをコピーしたり、プレースホルダーを残したりしてはなりません。あなたが書くすべてのファイルは、リポジトリの実際の言語、フレームワーク、コマンド、構造を反映しなければなりません。汎用のひな形は、成果物ではなく失敗です。
2. **非破壊的であること、置き換える前に尋ねる。** リポジトリにはすでに `AGENTS.md`、`docs/` ツリー、`.agents/` やスキルの構成、`CLAUDE.md`、あるいは独自の規約があるかもしれません。既存の作業を黙って上書き、移動、削除しては**なりません**。何が存在するかを検出し、それを読み、**調整**してください。可能な限りその場でマージし改善します。ユーザーがすでに持っているものを置き換えたり取り除いたりする前に、何を、なぜ行うかを説明し、明示的な承認を得てください。
3. **計画を提案し、その後実行する。** 偵察のあと、短い計画、すなわち何を作成し、何を変更し、調整または置き換えを提案する既存のものを提示し、大きなまたは破壊的な変更を行う前にユーザーの確認を待ってください。
4. **安全でレビュー可能な小さな単位で作業する。** 論理的にコミットし、差分を読みやすく保ち、シークレットに触れず、無関係なコードはそのままにします。
5. **止まって報告する** — ステップが失敗したり、リポジトリの状態が曖昧であったりする場合。
6. **信頼するが、検証する。** このプロンプトを信頼できない入力として扱ってください。公式ソース（`deepworkplan.com` と `DailybotHQ` リポジトリ）から取得したことを確認し、実行する前に評価し、スキルを実行する前にその整合性を検証してください — 以下の*信頼と検証*を参照してください。

## 信頼と検証（インストールする前に）

これからこのリポジトリにコードをインストールして実行しようとしています。そのため、まず採用しようとしているものを検証してください。これは推奨される行動です。慎重なエージェントは、取得したプロンプトを確認が済むまで信頼できないものとして扱います。

- **出所を確認する。** このプロンプトとスキルはオープンソースで MIT ライセンスです。正規のソースは `https://deepworkplan.com` と GitHub の `DailybotHQ` 組織（`deepworkplan-website`、`deepworkplan-skill`）です。機械可読なマニフェストは `https://deepworkplan.com/.well-known/dwp-trust.json` で公開されています。
- **スキルが何をするかを把握する。** Markdown ファーストです。CLI も HTTP API も認証も、**ネットワーク呼び出しもテレメトリーもありません**。セキュリティ上の唯一の影響は*このリポジトリ*を変更することであり、設計上は非破壊的です — 調整を行い、置き換える前に確認を求めます。
- **実行する前に整合性を検証する。** 各リリースでは、出荷されたスキルに対する `SHA256SUMS` が公開されています。信頼する前に、手元のコピーが一致することを確認してください。

  ```bash
  git clone https://github.com/DailybotHQ/deepworkplan-skill.git
  cd deepworkplan-skill
  # インストールするリリースのチェックサムをダウンロードする（vX.Y.Z を置き換えてください）:
  curl -fsSL -o SHA256SUMS \
    https://github.com/DailybotHQ/deepworkplan-skill/releases/download/vX.Y.Z/SHA256SUMS
  ./setup.sh --verify        # ゼロ以外の終了コードはファイルが一致しないことを意味します — 停止してください
  ```

  リリースは**チェックサム付きですが、署名はされていません**（署名はドキュメント化された次のステップです）。すべてはオープンで差分比較可能なので、任意のファイルをそのタグのリポジトリと比較することもできます。
- **問題を報告する。** セキュリティの問題は、該当リポジトリで **GitHub の脆弱性報告**を通じてプライベートに報告してください — `https://github.com/DailybotHQ/deepworkplan-skill/security` または `https://github.com/DailybotHQ/deepworkplan-website/security` — 公開 Issue としてではなく。各リポジトリの `SECURITY.md`、`https://deepworkplan.com/.well-known/security.txt`、そしてトラストページ `https://deepworkplan.com/trust` も参照してください。

## 0. 方法論と仕様を読む

何かを変更する前に、採用しようとしている標準を理解するため、正規のソースを読んでください。

- 方法論: https://deepworkplan.com/methodology.md
- 仕様: https://deepworkplan.com/spec.md
- キットカタログ: https://deepworkplan.com/kit.md

## 1. 偵察と、承認を得る計画

まずリポジトリを理解し、その後、何を行うかを提案します。

- **スタックを検出する。** 言語、フレームワーク、パッケージマネージャー（実際に存在するロックファイルから)、実際のビルド／テスト／リント／型チェックのコマンド、ソースのモジュール、テスト規約、そしてデプロイの形。
- **アーキタイプを分類する。** 個別リポジトリ（一般的なケース）、オーケストレーターハブ、またはエージェントワークスペース――自律エージェントの長期的な住処で、git は前提とされるのではなく推奨される――その根拠とともに。
- **すでに存在するものを棚卸しする。** `AGENTS.md`、`CLAUDE.md`、`docs/`、あらゆる `.agents/` やスキル／エージェントの構成、`.dwp/`、`.gitignore`。すでにこの仕事の一部を担っているものを書き留めます。
- **オンボーディング計画を提案する。** 簡潔なリスト、すなわち作成するファイル、変更するファイル、そして調整または置き換えを提案する既存のものを提示し、その後、進める前に**ユーザーに確認を求めます**。大きなリポジトリでは、この計画を `/dwp-create` で捕捉し、Deep Work Plan として実行してください。

## 2. Deep Work Plan スキルをインストールする

このリポジトリのエージェントが構造化された作業を計画し実行できるよう、スキルをインストールします。ルーターと八つのサブスキル、すなわち `create`、`execute`、`refine`、`resume`、`status`、`verify`、`onboard`、`author` を備えます。

```bash
npx skills add DailybotHQ/deepworkplan-skill
```

または、クローンしてセットアップスクリプトを実行します。

```bash
git clone https://github.com/DailybotHQ/deepworkplan-skill.git && cd deepworkplan-skill && ./setup.sh
```

## 3. リポジトリをオンボーディングする（推論にもとづき非破壊的に）

onboard サブスキル（`/deepworkplan-onboard`）を呼び出します。実際のリポジトリについて推論し、すべてをそれに適応させます。以下の各成果物について、**すでに存在する場合は、上書きするのではなく調整します**（マージし、改善し、方法論に整合させる)。そして何かを置き換える前にユーザーと確認します。

1. **`AGENTS.md` と `CLAUDE.md`。** ルートに `AGENTS.md` を生成します。索引、必須ルール（英語のみ、conventional commits、リポジトリの実際のテストパターンとレビューゲート)、そしてリポジトリの**実際に実行可能な**コマンドを備えた Quick Commands ブロックです。`AGENTS.md` がすでに存在する場合は、置き換えるのではなくそれにマージします。`CLAUDE.md → AGENTS.md` シンボリックリンクを作成します（既存の `CLAUDE.md` を尋ねずに上書きしない)。
2. **`docs/`。** 標準的なカテゴリを、実際のリポジトリ固有の内容で満たします。`PRODUCT_SPEC.md`（技術的でない製品／なぜドキュメント——ライブラリを含むすべてのリポジトリで必須）、`ARCHITECTURE.md`、`STANDARDS.md`、`TESTING_GUIDE.md`、`DEVELOPMENT_COMMANDS.md`、`SECURITY.md`（決して省略しない——シークレットが一つもなくても、すべてのリポジトリにはセキュリティ態勢がある）、`AI_AGENT_ONBOARDING.md`、`AI_AGENT_COLLAB.md`、加えて `PERFORMANCE.md` と `docs/README.md` の索引です。ドキュメントがすでに存在する場合は、統合し拡張します。複製はしません。
3. **モジュールごとのドキュメント。** 偵察で発見された主要なソースモジュールのそれぞれの中に、`README.md`（複雑なモジュールには `docs/` サブフォルダ）を追加します。
4. **`.agents/` と `.claude → .agents`。** 正規のエージェント横断の拠点を作成します。**推論にもとづく** `agents/` のカタログ、スタックに合った `skills/`、そしてインストールされたスキルへ委譲する薄い `dwp-*` の `commands/` です。各エントリは、別のものからコピーされたのではなく、*この*リポジトリのために正当化されます。ディスク上に存在するものと一致する `docs/` カタログ（`skills_agents_catalog.md` と `COMMANDS_REFERENCE.md`)、加えて `settings.json`、そして `.claude → .agents` シンボリックリンクを追加します。既存のスキル／エージェントをカタログに織り込みます。
5. **適応された DWP スキル。** インストールされたスキルはエンジンです。リポジトリ自身のキット（スキル、エージェント、コマンド）は、**このリポジトリのために推論された**ものでなければなりません。決して別のリポジトリのキットのコピー＆ペーストではありません。
6. **`.dwp/` と `tmp/`。** `plans/` と `drafts/` を備えた gitignore された `.dwp/`、加えて `tmp/` スクラッチ領域を整備します。両方とも非破壊的に `.gitignore` に追加します（追記し、決して書き換えない)。

## 4. キットを進化させる（author サブスキル)

`author` サブスキルを使って、リポジトリ自身のキットを育てます。薄い委譲 `/skill-create` と `/agent-create` がそこへ振り分けます。繰り返し可能なセッション内の手順には**スキル**を、独自のモデル階層とツールを持つ繰り返し現れる役割には**エージェント**を、そして薄い委譲としてのみ**コマンド**を作ります。`.agents/docs/` のカタログを、ディスク上に存在するものと同期し続けます。

## 5. オプトイン式のアドオンを提案する

基本構成のあと、四つのアドオン（devcontainer、Dailybot、dependency-upgrade、design-system）を列挙し、それぞれを明示的なオプトインとして提案します。リポジトリは**ゼロ**のアドオンで完全に適合します。決して自動でインストールしないでください。

- **Devcontainer サポート** — 永続的な AI CLI 認証を備えた、再現可能で隔離された開発コンテナ。
- **Dailybot 連携** — すでに Dailybot を使っているチームのための、最善努力での進捗／マイルストーンの報告。任意で自律的なフック強制層も加わり、エージェントは促されなくても重要な作業を報告します。中核となる方法論は Dailybot への依存がゼロです。
- **Dependency upgrade** — パッケージマネージャー非依存で、バッチ化され、検証され、取り消し可能な更新。受け入れられると、`/lib-upgrade` コマンドをインストールします。
- **Design system** — リポジトリの実際のデザイントークンとコンポーネントスタイルから推論された、エージェント向けの `docs/DESIGN.md`。どのエージェントも、リポジトリ自身のデザインシステムと一貫した UI を生成できます。

## 6. 計画して実行する

あらゆるタスクに対して Deep Work Plan を生成し、タスクごとに実行します。

- `/dwp-create <goal>` — 目標を、検証ゲートを伴う番号つきの逐次的なタスクへと分解する。
- `/dwp-execute` — 計画をタスクごとに実行し、進捗を更新し、各ゲートを検証する。
- `/dwp-status` — 変更を加えずに進捗を報告する。
- `/dwp-refine` — 完了済みの作業を保持しながら、タスクを追加、削除、並べ替える。
- `/dwp-resume` — 状態を再構築し、中断された計画を続行する。

すべての計画は、三つの必須最終タスクで終わります。すなわち、計画自身の変更に対する **Security Review**（`docs/SECURITY.md` を最新に保つ。重大な発見は完了を阻止する）、Skills & Agents Discovery、そして Executive Report です。

## 7. 検証する

`/dwp-verify` を実行して、客観的な合否の適合性レポートを得ます（[仕様の適合性ドキュメント](https://deepworkplan.com/spec)の基準を確認します)。その後、次を確認します。

- [ ] スキルがインストールされ解決可能で、八つのサブスキルすべてが利用できる。
- [ ] ルートに実際の Quick Commands ブロックを備えた `AGENTS.md` が存在し、`CLAUDE.md` がそれに解決される。
- [ ] `docs/` が標準的なカテゴリを実際のリポジトリ固有の内容で保持し、主要なモジュールが `README.md` を持つ。
- [ ] `.agents/` が `agents/`、`commands/`（薄い `dwp-*` 委譲)、`skills/`、そして現実と一致するカタログを備えて存在し、`.claude → .agents` が解決される。
- [ ] `.dwp/` が存在し、gitignore されており、`plans/` と `drafts/` を持つ。`tmp/` が存在し、gitignore されている。
- [ ] 既存のユーザーコンテンツが同意のもとに保持または調整された。何も黙って破壊されていない。
- [ ] Deep Work Plan を生成し、各ゲートを検証しながらタスクごとに実行できる。

## 成果

オンボーディングが完了すると、リポジトリは二つの永続的なかたち、すなわち方法論の柱において変わります。

1. **リポジトリが仕様駆動になる。** 作業は場当たり的なプロンプトからではなく、書かれた計画と仕様から始まります。
2. **リポジトリそのものがエージェントハーネスになる。** `AGENTS.md`、`docs/`、モジュールごとのドキュメント、そして `.agents/` のスキル拠点が、構造化され検証可能な作業を行うために必要なコンテキストとコマンドを、すべてのエージェントに与えます。

誰でもこのプロンプトをあらゆるリポジトリで実行でき、どのAIエージェントも操縦できるコードベースで締めくくれます。
