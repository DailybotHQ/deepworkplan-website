## Deep Work Plan 스펙

스펙은 방법론의 정밀하고 읽기 쉬운 정의입니다 — 사람과 에이전트가 공유하는 구조와 프로토콜입니다. 규범적 RFC-2119 용어로, 스펙 주도 계획이 어떻게 구조화되고 에이전트가 그것에 어떻게 맞춰 실행해야 하는지를 명시합니다: 계획이 진실 공급원이고, 검증 게이트는 이진(binary)이며, 리포지토리 자체가 에이전트에 필요한 하니스를 담습니다. 순서가 있는 문서들로 구성됩니다.

- **문서화 표준** — AI-first 리포지토리 구조.
- **DWP 스펙**(v1.2) — 계획 구조, 작업 구조, 실행 루프, 브라운필드 변경을 위한 Delta 절, DWP 재개 프로토콜, 비례적 엄격도 계층(micro/standard/deep), 기계 가독 계획 상태 레이어.
- **에이전트 프로토콜**(v1.2) — 요구되는 교차 에이전트 행동, 명령 매핑, 지원 에이전트(OpenClaw와 Hermes 포함), 실행 프로필(대화형 대 무인) — 정지 조건과 예약된 계속 실행.
- **아키타입**(v1.1) — 개별 리포지토리, 오케스트레이터 허브, 에이전트 작업 공간(자율 에이전트의 장기 실행 홈: OpenClaw 작업 공간, Hermes 서비스 디렉터리, 클라우드 에이전트 볼륨); 분류 휴리스틱과 온보딩 차이.
- **애드온** — 선택 기능을 얹는 선택형 메커니즘으로, author 하위 스킬(리포지토리가 자체 키트를 키움), dependency-upgrade 같은 유지보수 애드온, 그리고 design-system 애드온(리포지토리의 실제 디자인 소스에서 추론되고 비주얼 UI, CLI 출력, 대화형 표면을 위한 프로필을 갖춘 `docs/DESIGN.md`)을 포함합니다.
- **적합성** — AI-first 리포지토리의 규범적 정의: 리포지토리가 반드시 그리고 권장으로 갖춰야 하는 산출물, 무엇이 계획을 잘 구성되게 하는지, 그리고 `/dwp-verify`로 객관적으로 검증하는 방법.
- **계획 상태** — 기계 가독 상태 레이어: `manifest.json`과 `state.json`, 게이트 기록, 에피소드 메모리로서의 결과 기록, 조정(Markdown이 승리), 그리고 레이어가 필요한 시점.
