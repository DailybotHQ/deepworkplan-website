## ระเบียบวิธี Deep Work Plan

Deep Work Plan (DWP) คือระเบียบวิธีแบบเปิดที่เป็นกลางต่อเฟรมเวิร์กสำหรับการดำเนินงานวิศวกรรมอย่างมีโครงสร้างและอิสระด้วยเอเจนต์เขียนโค้ด AI มันเปลี่ยนเป้าหมายที่คลุมเครือให้กลายเป็นแผนที่ทบทวนได้ ซึ่งเอเจนต์สามารถดำเนินการ หยุดพัก ดำเนินการต่อ และรายงานผลได้ — โดยไม่สูญเสียบริบท

DWP ตั้งอยู่บนเสาหลักสองต้น

**spec-driven development** แทนที่จะถือว่าบทสนทนาแชตเป็นแหล่งความจริง คุณเขียนลงไปก่อนว่า *อะไร* ควรเป็นจริง — เป้าหมาย ขอบเขต เกณฑ์การยอมรับ และการตรวจสอบที่พิสูจน์ว่าเสร็จ — และเอเจนต์ดำเนินการเทียบกับข้อกำหนดนั้น ใน DWP แผน *คือ* ข้อกำหนด แผน → atomic task → validation gate → การเสร็จสิ้น การเขียนข้อกำหนดก่อนลดการเบี่ยงเบน (เอเจนต์ถูกวัดเทียบกับเกณฑ์ที่ระบุไว้) ทำให้งานตรวจสอบได้ (ทุก gate ผ่านหรือไม่ผ่าน) และทำให้ดำเนินต่อได้ (ข้อกำหนดอยู่ได้นานกว่าเซสชันหรือเอเจนต์ใด) ด้วยความที่เป็นกลางต่อเครื่องมือและฝังอยู่ใน repository โดยกำเนิด มันจึงต่างจากระบบขับเคลื่อนด้วยข้อกำหนดที่ผูกติดกับเครื่องมืออย่าง GitHub Spec Kit, Amazon Kiro และ Tessl

**harness engineering — repository กลายเป็น harness** ตัวแบบจำลองภาษาโดยลำพังเป็นเพียงตัวทำนายข้อความ สิ่งที่ทำให้มันเป็นวิศวกรที่เชื่อถือได้คือ *harness* รอบตัวมัน — บริบท เครื่องมือ control loop guardrail และสถานะถาวร harness engineering คือการออกแบบโครงนั่งร้านนั้นอย่างจงใจ จุดยืนของ DWP คือมันควรอยู่ใน repository ไม่ใช่ในเครื่องมือ ได้แก่ บริบทในรูป `AGENTS.md` + `docs/` เครื่องมือในรูป kit `.agents/` control loop ในรูปแผน guardrail ในรูป validation gate และสถานะในรูป `.dwp/` ที่ถูก gitignore เพราะทุกส่วนเป็นไฟล์ใน repo harness จึงพกพาได้ — เอเจนต์ใดก็ขับเคลื่อน repo ใดก็ได้

ระเบียบวิธีอธิบายไว้ในห้าบท

1. **แถลงการณ์** — Deep Work Plan คืออะไรและทำไมการดำเนินงานแบบมีโครงสร้างจึงสำคัญ
2. **ลูปหลัก** — วงจรวางแผน-ดำเนินการ-ตรวจสอบที่ขับเคลื่อนทุกงาน
3. **เทมเพลต** — โครงสร้างแผนและงานที่เอเจนต์และมนุษย์ใช้ร่วมกัน
4. **Skills และ agents** — วิธีที่คำสั่งและบทบาทของ agent ทำงานร่วมกัน kit ที่มีชีวิต (วิธีที่ repository ปลูก skill ของตัวเองผ่าน sub-skill ชื่อ author) และส่วนเสริมแบบเลือกเข้าร่วมอย่าง dependency-upgrade และส่วนเสริม design-system ฝั่ง frontend
5. **Archetype** — การปรับระเบียบวิธีให้เข้ากับ repo เดี่ยวและ orchestrator hub
