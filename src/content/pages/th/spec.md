## ข้อกำหนด Deep Work Plan

ข้อกำหนดคือนิยามที่แม่นยำและอ่านได้ของระเบียบวิธี — โครงสร้างและโปรโตคอลที่มนุษย์และเอเจนต์ใช้ร่วมกัน มันระบุในเชิงบรรทัดฐานตามถ้อยคำแบบ RFC-2119 ว่าแผนที่ขับเคลื่อนด้วยข้อกำหนดมีโครงสร้างอย่างไรและเอเจนต์ต้องดำเนินการเทียบกับมันอย่างไร แผนคือแหล่งความจริง validation gate เป็นแบบสองค่า และ repository เองพก harness ที่เอเจนต์ต้องการ มันถูกจัดเป็นเอกสารที่เรียงลำดับ

- **มาตรฐานเอกสาร** — โครงสร้าง repository แบบ AI-first
- **ข้อกำหนด DWP** (v1.2) — โครงสร้างแผน โครงสร้างงาน ลูปดำเนินงาน ส่วน Delta สำหรับการเปลี่ยนแปลง brownfield DWP Resume Protocol ระดับความเข้มงวดตามสัดส่วน (micro/standard/deep) และชั้นสถานะแผนที่เครื่องอ่านได้
- **โปรโตคอลของเอเจนต์** (v1.2) — พฤติกรรมข้ามเอเจนต์ที่จำเป็น การจับคู่คำสั่ง เอเจนต์ที่รองรับ (รวมถึง OpenClaw และ Hermes) และโปรไฟล์การดำเนินงาน (แบบโต้ตอบเทียบกับแบบไม่มีผู้ดูแล) พร้อมเงื่อนไขหยุดและ scheduled continuation
- **Archetype** (v1.1) — repo เดี่ยว orchestrator hub และ agent workspace (บ้านที่มีอายุยาวนานของเอเจนต์อัตโนมัติ: OpenClaw workspace ไดเรกทอรีบริการ Hermes persistent volume ของ cloud agent); หลักการจำแนกประเภทและความแตกต่างของการออนบอร์ด
- **ส่วนเสริม** — กลไกแบบเลือกเข้าร่วมสำหรับวางซ้อนความสามารถที่เป็นทางเลือก รวมถึง sub-skill ชื่อ author (เพื่อให้ repository ปลูก kit ของตัวเอง) ส่วนเสริมการบำรุงรักษาอย่าง dependency-upgrade และส่วนเสริม design-system (`docs/DESIGN.md` ที่ให้เหตุผลจากแหล่งดีไซน์จริงของ repo พร้อมโปรไฟล์สำหรับ UI เชิงภาพ เอาต์พุต CLI และพื้นผิวเชิงสนทนา)
- **ความสอดคล้อง** — นิยามเชิงบรรทัดฐานของ repository แบบ AI-first: สิ่งประดิษฐ์ที่ repository MUST และ SHOULD มี สิ่งที่ทำให้แผนมีรูปแบบที่ถูกต้อง และวิธีตรวจสอบมันอย่างเป็นวัตถุวิสัยด้วย `/dwp-verify`
- **สถานะแผน** — ชั้นสถานะที่เครื่องอ่านได้: `manifest.json` และ `state.json` บันทึก gate บันทึกผลลัพธ์ในฐานะหน่วยความจำเชิงลำดับเหตุการณ์ การปรับประสานงาน (markdown ชนะ) และเมื่อใดที่ชั้นนี้จำเป็น
