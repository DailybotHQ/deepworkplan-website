# Адаптер хмарних агентів

Хмарні та фонові агенти — віддалені завдання Claude Code, OpenAI Codex cloud, агенти класу Jules та подібні асинхронні середовища виконання — є ефемерними за своєю природою: сесія починається, виконується протягом обмеженого часу й завершується. DWP перетворює це обмеження на перевагу, роблячи план довговічним артефактом, а не сесію.

## Ключова ідея

Каталог `.dwp/` репозиторію є **довговічним рівнем специфікації та стану**. Хмарні сесії є **ефемерними виконавцями**. План повідомляє виконавцю, що робити; `state.json` повідомляє, де відновити роботу. Сесія може бути будь-якою — інша модель, інший провайдер, новий контейнер — і план продовжиться з рівно того місця, де зупинився.

## Рівень підтримки

**Повний** — будь-який хмарний агент, що читає репозиторій, дотримується `AGENTS.md` та може виконувати команди оболонки, підтримує DWP без адаптера під конкретну платформу.

## Цикл прокидання-відновлення-поступки

Кожен поворот хмарного агента слідує паттерну:

1. **Прокидання** — сесія починається (ініційована cron-розкладом, подією CI або дією користувача).
2. **Запуск Протоколу відновлення DWP** — читання README плану, пошук checkpoint, узгодження `state.json` із markdown, огляд шва, запуск диму-тесту. Див. [Специфікацію DWP](/spec/dwp-specification#the-dwp-resume-protocol).
3. **Перевірка заблокованого стану** — якщо `state.json.blocked` встановлено, звітувати про причину й поступитися без виконання.
4. **Виконання наступного атомарного завдання** — одне завдання, з повним проходженням його валідаційного gate.
5. **Оновлення рівня стану** — атомарний перезапис `state.json` із новим статусом завдання, записами gate, записом результату та хешем коміту.
6. **Поступка** — сесія завершується чисто; наступний поворот продовжує з оновленого стану.

## Вимоги

- План MUST нести машиночитаний рівень стану (`manifest.json` + `state.json`). Без нього ефемерна сесія не має надійного способу визначити позицію відновлення. Див. [Стан плану](/spec/plan-state).
- План MUST бути схвалений людиною перед будь-яким автономним хмарним поворотом. Схвалення плану є контрольною точкою.
- Агент діє в межах обмежених повноважень: він MUST NOT розширювати scope, виконувати деструктивні або зовнішньоспрямовані дії, що план явно не авторизує, та MUST NOT розтягувати інструкції завдання для охоплення виявленої, але незапланованої роботи — виявлена робота фіксується для наступного `refine`, а не імпровізується. Див. [Протокол агента](/spec/agent-protocol#execution-profiles).

## Нотатки

Хмарні агентні сесії є граничним випадком профілю автономного виконання: нульова неперервність сесії, максимальна довговічність плану. Каталог `.dwp/` DWP спеціально розроблений для забезпечення цього — весь стан, потрібний агенту для початку, продовження та завершення обсягу роботи, зберігається у звичайних markdown і JSON-файлах, які будь-яка сесія може прочитати зі свіжого checkout.
